# CalendarProvider

`CalendarProvider` is the required outer wrapper for every Big Calendar. Think of it as the brain of the whole calendar — it holds the active view, the focus date, the event list, and all the settings. Every component you place inside it (the toolbar, the grid, your own custom controls) reads from that shared brain automatically.

You never need to pass events or navigation state down through props by hand. Everything flows through the provider.

```tsx
import { CalendarProvider, Calendar } from '@big-calendar/react'
import { createTemporalLocalizer } from '@big-calendar/localizer-temporal'
import '@big-calendar/styles/index.css'

const localizer = await createTemporalLocalizer({ locale: 'en-US', timeZone: 'UTC' })

function App() {
  return (
    <CalendarProvider localizer={localizer} events={myEvents} defaultDate="2026-06-15">
      <Calendar />
    </CalendarProvider>
  )
}
```

## Data props

These props feed the calendar its content. You own the data — the calendar reads it and fires callbacks, but never modifies it directly.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>events</code></td>
      <td><code>TEvent[]</code></td>
      <td><code>[]</code></td>
      <td>
        Your foreground event objects. Each one needs at minimum a start date, an end date, and
        a title — but you can use any field names you like and tell the calendar about them via
        <code>accessors</code>. The calendar renders these as coloured blocks across the views.
      </td>
    </tr>
    <tr>
      <td><code>backgroundEvents</code></td>
      <td><code>TEvent[]</code></td>
      <td><code>[]</code></td>
      <td>
        Background events render as coloured bands behind the foreground events. Clicks and
        drags pass straight through them — they are visual only. Useful for showing
        availability windows, focus blocks, or blocked-off periods. See
        <LinkTo title="Background Events/Overview" story="docs">Background Events</LinkTo>.
      </td>
    </tr>
    <tr>
      <td><code>resources</code></td>
      <td><code>TResource[]</code></td>
      <td><em>none</em></td>
      <td>
        Resource objects for multi-resource time grids (e.g. rooms or staff). When supplied,
        each day column in the time grid is split into one sub-column per resource. See
        <LinkTo title="Resources/Overview" story="docs">Resources</LinkTo>.
      </td>
    </tr>
  </tbody>
</table>

## Date and view

### Uncontrolled (the calendar manages its own state)

Pass these to set the initial values and then let the calendar handle navigation on its own. Most apps start here.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>defaultView</code></td>
      <td><code>ViewKey</code></td>
      <td><code>'month'</code></td>
      <td>
        Which view the calendar opens on. Valid values: <code>'month'</code>,
        <code>'week'</code>, <code>'work_week'</code>, <code>'day'</code>,
        <code>'agenda'</code>. The user can switch views with the toolbar after that.
      </td>
    </tr>
    <tr>
      <td><code>defaultDate</code></td>
      <td><code>string</code></td>
      <td>Today</td>
      <td>
        The date the calendar focuses on when it first loads, as an ISO 8601 string
        (e.g. <code>"2026-06-15"</code>). The user can navigate away from it with Prev / Next.
      </td>
    </tr>
  </tbody>
</table>

### Controlled (your state drives the calendar)

Pass these when your application needs to programmatically control which view or date is shown — for example, a custom mini-calendar or a URL-driven route.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>view</code></td>
      <td><code>ViewKey</code></td>
      <td>
        The active view. When you pass this, the calendar will always show whatever view your
        state holds. Wire <code>onView</code> to update your state when the user switches views
        via the toolbar.
      </td>
    </tr>
    <tr>
      <td><code>date</code></td>
      <td><code>string</code></td>
      <td>
        The focus date as an ISO 8601 string. Wire <code>onNavigate</code> to update your state
        when the user navigates with the toolbar.
      </td>
    </tr>
  </tbody>
</table>

```tsx
// Fully controlled view + date:
const [view, setView] = useState('week')
const [date, setDate] = useState('2026-06-15')

<CalendarProvider
  view={view}
  date={date}
  onView={({ view }) => setView(view)}
  onNavigate={({ date }) => setDate(date)}
  ...
>
```

## Time grid props

These props control the Week, Work Week, and Day views. They have no effect on the Month or Agenda view.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>step</code></td>
      <td><code>number</code></td>
      <td><code>30</code></td>
      <td>
        The size of each time slot in minutes. <code>30</code> means slots are "9:00", "9:30",
        "10:00", and so on. Set to <code>15</code> for quarter-hour slots or <code>60</code>
        for one-hour slots.
      </td>
    </tr>
    <tr>
      <td><code>timeslots</code></td>
      <td><code>number</code></td>
      <td><code>2</code></td>
      <td>
        How many individual slot rows appear under each labelled time. At <code>step=30</code>
        and <code>timeslots=2</code> you get two rows per half-hour (a light divider line
        splits each block). Setting this to <code>1</code> removes the subdivisions.
      </td>
    </tr>
    <tr>
      <td><code>min</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>Start of day</td>
      <td>
        The earliest time shown in the time grid. <code>hour</code> is 0–23;
        <code>minute</code> defaults to <code>0</code> when omitted. For example,
        <code>&#123; hour: 8 &#125;</code> starts the grid at 8:00 AM. Events that begin
        before this time are still rendered; they are clipped at the top.
      </td>
    </tr>
    <tr>
      <td><code>max</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>End of day</td>
      <td>
        The latest time shown. <code>&#123; hour: 23 &#125;</code> ends the grid at
        11:00 PM — all slots up to (but not including) that boundary are visible.
        <code>&#123; hour: 0 &#125;</code> means end of day (full 24-hour grid).
        Events that run past this time are clipped at the bottom.
      </td>
    </tr>
    <tr>
      <td><code>scrollToTime</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>Current time</td>
      <td>
        The wall-clock time the time grid scrolls to when it first loads.
        When omitted, the grid scrolls to the current time in the calendar's
        configured time zone — keeping "now" in view without any extra work.
        Pass a fixed value like <code>&#123; hour: 8 &#125;</code> to always open at
        8:00 AM regardless of the time of day. The value is read once on mount;
        changing it after the calendar has loaded has no effect.
      </td>
    </tr>
    <tr>
      <td><code>dayLayoutAlgorithm</code></td>
      <td><code>'overlap' | 'no-overlap' | Function</code></td>
      <td><code>'overlap'</code></td>
      <td>
        Controls how events that share the same time slot are packed side-by-side in a day column.
        <ul>
          <li>
            <code>'overlap'</code> — groups events that start close together and lets them
            overlap, sharing the full column width. Looks like Google Calendar.
          </li>
          <li>
            <code>'no-overlap'</code> — gives every time-touching event its own lane so
            nothing overlaps visually. Uses more horizontal space but every event is
            fully readable.
          </li>
        </ul>
        You can also pass a custom function that receives the events and returns layout boxes.
      </td>
    </tr>
    <tr>
      <td><code>showAllEvents</code></td>
      <td><code>boolean</code></td>
      <td><code>false</code></td>
      <td>
        The all-day strip above the time grid is capped at two rows by default. When more
        all-day or multi-day events exist, a <strong>+ N more</strong> indicator appears.
        Set <code>showAllEvents</code> to <code>true</code> to remove that cap and show every
        event regardless of how tall the strip grows.
      </td>
    </tr>
    <tr>
      <td><code>showMultiDayTimes</code></td>
      <td><code>boolean</code></td>
      <td><code>false</code></td>
      <td>
        By default, events that span more than one day appear in the all-day strip. Set this
        to <code>true</code> to render them in the time columns instead, positioned at their
        actual start and end times. Useful for events like overnight shifts.
      </td>
    </tr>
  </tbody>
</table>

## Drilldown

"Drilling down" is what happens when a user clicks a day number in the month grid or a column heading in the week grid — the calendar zooms in to a single-day view for that date.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>drilldownView</code></td>
      <td><code>ViewKey | null</code></td>
      <td><code>'day'</code></td>
      <td>
        Which view to land on after a drilldown. Pass <code>null</code> to disable drilldown
        entirely — day numbers become plain text with no click handler.
      </td>
    </tr>
    <tr>
      <td><code>getDrilldownView</code></td>
      <td><code>Function</code></td>
      <td><em>none</em></td>
      <td>
        A function called with the clicked date and the current view. Return a view key to
        navigate there, or <code>null</code> to block the drilldown for that specific date.
        Overrides <code>drilldownView</code> when supplied.
      </td>
    </tr>
    <tr>
      <td><code>onDrillDown</code></td>
      <td><code>Function</code></td>
      <td><em>none</em></td>
      <td>
        When supplied, the calendar hands full control of drilldown to you — it will
        <em>not</em> change the view or date itself. Receives <code>{"{ date, view }"}</code>:
        the target date and the view the calendar would have navigated to. Use this to drive
        your own router instead of letting the calendar navigate.
      </td>
    </tr>
  </tbody>
</table>

## Selection

Slot selection is how users interact with empty time to indicate they want to create a new event. It is disabled by default.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>selectable</code></td>
      <td><code>boolean | 'ignoreEvents'</code></td>
      <td><code>false</code></td>
      <td>
        Enables slot selection. <code>true</code> lets users click and drag anywhere in the
        grid — including on top of existing events — to make a selection.
        <code>'ignoreEvents'</code> enables selection but prevents clicks directly on event
        blocks from starting a selection (those clicks fall through to the event click handler
        instead).
      </td>
    </tr>
    <tr>
      <td><code>longPressThreshold</code></td>
      <td><code>number</code></td>
      <td><code>500</code></td>
      <td>
        On touch screens, how long (in milliseconds) a finger must be held still before a
        slot selection begins. A shorter touch is treated as a tap. Mouse and stylus inputs
        are not affected. Decrease this number if the long-press feels slow; set it to
        <code>0</code> to start selections immediately on touch.
      </td>
    </tr>
  </tbody>
</table>

For the full slot callback reference and example payloads see <LinkTo title="Selection/Overview" story="docs">Selection → Overview</LinkTo>.

## Drag and drop

Drag-and-drop requires the optional `@big-calendar/dnd` package. Install it separately and wrap your `<Calendar>` with `useCalendarDnd`. These props control which events can be moved or resized and what happens when they are.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>draggableAccessor</code></td>
      <td><code>string | Function</code></td>
      <td>All events</td>
      <td>
        Controls which events can be dragged. Pass a string to read a boolean field from each
        event object, or a function <code>(event) =&gt; boolean</code> for custom logic. When
        omitted, every event is draggable. Return <code>false</code> to lock an event in place.
      </td>
    </tr>
    <tr>
      <td><code>resizableAccessor</code></td>
      <td><code>string | Function</code></td>
      <td>All events</td>
      <td>
        Same as <code>draggableAccessor</code> but controls whether an event shows resize
        handles. Return <code>false</code> to prevent resizing a specific event.
      </td>
    </tr>
  </tbody>
</table>

## Accessors

By default, Big Calendar looks for `title`, `start`, `end`, `allDay`, and `resource` fields on each event object. If your data uses different field names, supply an `accessors` object to map them.

<table>
  <thead><tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>accessors</code></td>
      <td><code>Partial&lt;Accessors&gt;</code></td>
      <td>Standard field names</td>
      <td>
        Override any accessor with either a string (field name to read) or a function
        <code>(event) =&gt; value</code> for computed values. You only need to supply the
        ones that differ from the defaults.
      </td>
    </tr>
  </tbody>
</table>

```tsx
// Your events use 'name' and 'startsAt' / 'endsAt' instead of the defaults:
<CalendarProvider
  accessors={{
    title: 'name',
    start: (event) => event.startsAt,
    end:   (event) => event.endsAt,
  }}
  ...
>
```

The full list of overridable accessors: `title`, `start`, `end`, `allDay`, `resource`, `id`.

## Callbacks

### Event interaction

These fire when a user interacts with an **existing event block**. They work on every view and do not require `selectable`.

<table>

<table>
