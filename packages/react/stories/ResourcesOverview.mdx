# Resources

Give the calendar a `resources` array and the **day** and **week** views automatically split into per-resource columns. A "rooms" schedule, a "people" calendar, a "lanes" planner — any list of named entities works.

## Basic setup

```tsx
const rooms = [
  { id: 'board', title: 'Board room' },
  { id: 'training', title: 'Training room' },
  { id: 'mtg1', title: 'Meeting room 1' },
]

<CalendarProvider
  resources={rooms}
  events={events}
  ...
>
```

Events route to their resource column via the `resource` accessor, which reads `event.resourceId` by default. Change the field name with `resourceIdAccessor`, or pass a function:

```tsx
// Field name shorthand:
<CalendarProvider resourceIdAccessor="roomId" ...>

// Function accessor:
<CalendarProvider resourceIdAccessor={(e) => e.room.id} ...>
```

Similarly, `resourceTitleAccessor` controls which field becomes the column header (defaults to `resource.title`).

## Layout modes

Two layout orderings are available.

### Resource-major (default)

All days for one resource appear together, then all days for the next:

```
[ Board room: Mon Tue Wed Thu Fri ] [ Training: Mon Tue Wed Thu Fri ]
```

```tsx
<CalendarProvider resources={rooms} defaultView="week" ...>
```

### Day-major

All resources for a day appear together, then the next day:

```
[ Monday: Board room  Training  Mtg1 ] [ Tuesday: Board room  Training  Mtg1 ]
```

```tsx
<CalendarProvider resources={rooms} resourceLayout="day" defaultView="week" ...>
```

Day-major is useful when you want to compare all resources' availability within a single day at a glance.

## Selection with resources

When `selectable` is on, the slot callbacks (`onSlotClick`, `onSlotDoubleClick`, `onSlotSelect`) also report the `resourceId` of the column the user clicked or dragged in. Use it to pre-fill the resource when opening a "create event" dialog:

```tsx
<CalendarProvider
  resources={rooms}
  selectable
  onSlotSelect={({ start, end, allDay, resourceId }) => {
    openCreateDialog({ start, end, allDay, resourceId })
  }}
>
```

## Drag and drop with resources

Move and resize work exactly the same as in the standard calendar. The difference is that `onEventDrop` and `onEventResize` now also report the `resourceId` of the column the event landed in:

```tsx
onEventDrop={({ event, start, end, allDay, resourceId }) => {
  setEvents(prev =>
    prev.map(e =>
      e.id === event.id
        ? { ...e, start, end, allDay, resourceId }  // reassign to the new resource
        : e
    )
  )
}}
```

Dragging an event into a **different** resource's column is a real edit: the `resourceId` changes. Your update assigns the event to its new resource — Big Calendar just reports it.

`onDropFromOutside` and `onEventResize` also include `resourceId` when resources are active.

## Events without a resource

Events that have no `resourceId` (or whose `resourceId` does not match any resource in the `resources` array) are not rendered in resource views. They simply do not appear — they are not dropped or hidden. Make sure every event has a valid `resourceId` when `resources` is set, or filter your event list accordingly.
