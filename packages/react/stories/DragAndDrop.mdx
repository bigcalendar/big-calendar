# Drag and drop — moving events

Dragging an event to a new day (or, later, a new time) is really an **edit** of that
event. Big Calendar splits this into clear jobs so each part stays simple:

- **`@big-calendar/core`** does the **date-math** — given where you dropped the event,
  it works out the event's new start/end (keeping its duration).
- **`@big-calendar/dnd`** does the **pointer work** — it turns the browser drag into
  "this event was dropped here" using the optional
  [Pragmatic Drag and Drop](https://atlassian.design/components/pragmatic-drag-and-drop/about)
  library.
- **You** do the **saving** — the calendar tells you the proposed change; your code
  writes it to your data (and to your server).

This is the same idea as a controlled `<input>`: the calendar shows what you give it
and tells you what the user did, but **you own the data**.

## Turning it on

Install the optional package and call the hook with a ref around your calendar:

```bash
pnpm add @big-calendar/dnd
```

```tsx
import { useRef } from 'react'
import { Calendar, CalendarProvider } from '@big-calendar/react'
import { useCalendarDnd } from '@big-calendar/react/dnd'  // dedicated DnD entry

function DraggableCalendar() {
  const ref = useRef(null)
  useCalendarDnd(ref) // enables drag-to-move for everything inside `ref`
  return (
    <div ref={ref}>
      <Calendar />
    </div>
  )
}
```

`useCalendarDnd` is exported from the dedicated **`@big-calendar/react/dnd`** subpath so
apps that don't use DnD can exclude it cleanly. It is also re-exported from the main
`@big-calendar/react` barrel for convenience — both imports work.

`useCalendarDnd` must be **inside** a `CalendarProvider`. It enables moving in the
**month** view (whole-day shifts) and the **time-grid** week / day / work-week views
(the start snaps to the dropped slot, keeping the event's duration). The agenda view has
no move.

## What `onEventDrop` gives you

When the user drops an event, the calendar calls `onEventDrop` with the **original**
event plus the **proposed** new bounds:

```ts
onEventDrop={({ event, start, end, allDay }) => {
  // `event`            → the original event object, untouched
  // `start`/`end`      → the proposed new times, as ISO strings
  // `allDay`           → whether it's now a whole-day event
}}
```

Two things to notice, both on purpose:

1. **It's a report, not a change.** The calendar never edits your `events`. It can't,
   really — it reads your events through *accessors* (which are read-only), so it has no
   way to write `start`/`end` back into your particular event shape. You do that.
2. **You get the original event**, so undoing a change is trivial.

## The simple case

If you don't need a server round-trip, just apply the new bounds to your state:

```tsx
const [events, setEvents] = useState(initialEvents)

<CalendarProvider events={events}
  onEventDrop={({ event, start, end, allDay }) =>
    setEvents(prev =>
      prev.map(e => (e.id === event.id ? { ...e, start, end, allDay } : e)),
    )
  }
>
```

## The real case: async save with rollback

Usually a move has to be saved to a server, and that can fail. The standard pattern is
**optimistic update + rollback**: move the event right away so the UI feels instant,
then save; if the save fails, put it back.

```tsx
const [events, setEvents] = useState(initialEvents)

const onEventDrop = ({ event, start, end, allDay }) => {
  const previous = events // remember the state to roll back to

  // 1. optimistic update — show the move immediately
  setEvents(prev =>
    prev.map(e => (e.id === event.id ? { ...e, start, end, allDay } : e)),
  )

  // 2. save; the calendar does NOT await this — the lifecycle is yours
  saveEvent(event.id, { start, end, allDay }).catch(() => {
    // 3. on failure, roll back to the original state
    setEvents(previous)
  })
}
```

Your handler can be `async`. The calendar never waits on it and never holds a copy of
your data, so there is exactly one source of truth: your `events` state. See the
**Drag and drop → AsyncSaveWithRollback** story for a runnable version (with a checkbox
that forces the save to fail).

## Resizing

In the **time-grid** views, hovering a timed event reveals a grab bar on its top and
bottom edge. Dragging an edge to another slot resizes the event: the dragged edge snaps
to the slot, the other edge stays put, and (like move) you can drag past a day boundary
to grow the event across midnight. The event is clamped to a one-slot minimum so it can
never collapse. While you drag, a dashed **live preview** box shows the proposed new
size so you're not guessing from the pointer alone.

In the **month** grid the same idea applies by whole days: hovering a segment reveals a
grab strip on its **leading** and **trailing** edge, and dragging an edge to another day
grows or shrinks the event by whole days (the time of day is kept; the event can't drop
below one day). If an event spans more than one week row, the leading handle sits on its
first row and the trailing handle on its last — and dragging an edge across the week
boundary just works, because the drop target is an absolute day. A highlighted day-cell
band previews the proposed span.

The same dashed/highlighted preview also follows a **move** (not just a resize), so you
can see where an event will land before you let go.

Resize reports through `onEventResize`, which has the **exact same shape** as
`onEventDrop` — so apply (and roll back) it the same way:

```tsx
// Both reports are { event, start, end, allDay } — one handler covers them.
const apply = ({ event, start, end, allDay }) =>
  setEvents(prev => prev.map(e => (e.id === event.id ? { ...e, start, end, allDay } : e)))

<CalendarProvider events={events} onEventDrop={apply} onEventResize={apply}>
```

See the **Drag and drop → WeekEventResize** story for a runnable version.

## Dropping items from outside the calendar

You can drag something from **outside** the calendar — a palette of "event
templates", an unscheduled-task list — onto a time-grid slot to create a new
event there. A drop is a **create**, so it reports through a new callback,
`onDropFromOutside`, and (like every other report) *you* add the event:

```tsx
const [events, setEvents] = useState(initialEvents)

<CalendarProvider events={events}
  onDropFromOutside={({ start, end, allDay }) =>
    setEvents(prev => [...prev, { id: crypto.randomUUID(), title: 'New event', start, end, allDay }])
  }
>
```

The dropped item describes itself with a small **payload** — its
`durationMinutes` (and optionally `allDay`) — which core turns into the new
event's `start`/`end` at the dropped slot. There are two ways to carry it,
depending on what your drag source is:

- **A native `draggable="true"` element.** Set the payload as JSON on the
  `EXTERNAL_MIME` data type in `onDragStart`:

  ```tsx
  import { EXTERNAL_MIME } from '@big-calendar/dnd'

  <div draggable
    onDragStart={(e) => e.dataTransfer.setData(EXTERNAL_MIME, JSON.stringify({ durationMinutes: 60 }))}
  >1-hour focus block</div>
  ```

- **A [Pragmatic](https://atlassian.design/components/pragmatic-drag-and-drop/about)
  `draggable`.** Put the payload under the `EXTERNAL_DATA_KEY` key in
  `getInitialData` (`{ bcExternal: { durationMinutes: 60 } }`).

One difference between them matters for feedback: a **native** drag keeps its
payload hidden until the drop (a browser security rule), so the live preview can
only show a **single landing slot** while you drag — the real duration appears
when you release. A **Pragmatic** drag exposes its payload throughout, so the
preview shows the event's full proposed size as you move. Either way, the drop
itself is exact. See the **Drag and drop → DropFromOutside** story.

### Dropping onto the month grid

You can also drop onto the **month** grid, where a day cell has no time of its own,
so the payload decides what gets created:

- a payload with **no** `start`/`end` becomes a **whole-day** event on the dropped
  day (a `durationMinutes` is ignored here — there's no slot to place it in);
- a payload carrying a `start`/`end` **template** keeps its time-of-day and is
  **re-dated** to the dropped day — a "9–10am standup" template dropped on the 14th
  becomes 9–10am on the 14th, preserving the duration.

A highlighted day-cell band previews where the new event will land. See the
**Drag and drop → MonthDropFromOutside** story.

## Dragging events out of the calendar

The reverse also works: drag an event **out** to a dropzone you own (an
"unschedule" bin, a different list). Every event automatically exposes its data
on the native `dataTransfer` under the `EVENT_MIME` type, so a plain HTML5
dropzone can read it — no calendar code on the receiving side:

```tsx
import { EVENT_MIME } from '@big-calendar/dnd'

<div
  onDragOver={(e) => e.preventDefault()}
  onDrop={(e) => {
    const { id, title } = JSON.parse(e.dataTransfer.getData(EVENT_MIME))
    setEvents(prev => prev.filter(ev => String(ev.id) !== id))
  }}
>🗑 Unschedule</div>
```

`onEventDragStart` fires as the drag begins (it receives the event), handy for a
status line or to highlight valid drop zones. See the **Drag and drop →
DragOutToUnschedule** story.

## Keyboard drag and drop

Everything above works **without a mouse**, so the calendar stays usable for
keyboard and screen-reader users. Tab to an event, press **Space** to pick it up
(again to drop), and use the arrows — the step sizes match the surface:

**Time-grid** (week / day):

| Key | Action |
| --- | --- |
| **Space** | pick the event up (and again to drop) |
| **↑ / ↓** | move it one slot earlier / later |
| **← / →** | move it to the previous / next day |
| **Shift + ↑ / ↓** | shrink / grow the **end** edge by one slot |
| **Shift + Alt + ↑ / ↓** | grow / shrink the **start** edge by one slot |
| **Enter** / **Escape** | drop / cancel |

**Month** (whole days):

| Key | Action |
| --- | --- |
| **Space** | pick the event up (and again to drop) |
| **← / →** | move it ±1 day |
| **↑ / ↓** | move it ±1 week |
| **Shift + ← / →** | shrink / grow the **end** edge by a day |
| **Shift + ↑ / ↓** | shrink / grow the **end** edge by a week |
| **Shift + Alt + ← / →** | grow / shrink the **start** edge by a day |
| **Shift + Alt + ↑ / ↓** | grow / shrink the **start** edge by a week |
| **Enter** / **Escape** | drop / cancel |

While the event is "in hand" it dims and a preview shows where it will land — a
dashed box in the time grid, a highlighted day-cell band in the month grid — and
every step is announced through a polite live region (e.g. _"Standup, Tue 16
9:30 AM to 10:30 AM"_ in the time grid, _"Standup, Jun 16"_ in the month).
Dropping fires the **same** `onEventDrop` / `onEventResize` you already wired for
the mouse, so there's nothing extra to handle. (Enter still **opens** an event;
Space is what picks it up.) Keyboard drag is wired on the **month and time-grid**
views.

## Resource columns and cross-resource drag

Give the calendar a `resources` array and the **day** and **week** views
automatically split into per-resource columns — a "rooms" / "people" schedule.
The day view gives each resource one column under a single-tier title header; the
week view gives each resource a **two-tier** header (its title spanning that
resource's day columns, with the day names beneath) and lays the columns out
**resource-major** (all of one resource's days, then the next resource). Each
event lands in its resource's column via the `resource` accessor (which reads
`resourceId` by default), and each resource gets its own all-day lane:

```tsx
<CalendarProvider
  events={events}
  resources={[{ id: 'board', title: 'Board room' }, { id: 'training', title: 'Training room' }]}
  // event.resourceId decides the column; override with `resourceAccessor` if your field differs
/>
```

Dragging an event into a **different** resource's column is a real edit, so the
move/resize/drop callbacks now also report the **landing** `resourceId` — the
column you dropped onto:

```tsx
onEventDrop={({ event, start, end, allDay, resourceId }) => {
  // `resourceId` is the column dropped onto (undefined in a resource-less grid).
  // It's a report: you reassign the event to that resource in your own data.
  setEvents(prev => prev.map(e => (e.id === event.id ? { ...e, start, end, allDay, resourceId } : e)))
}}
```

The same `resourceId` rides along on `onEventResize`, `onDropFromOutside`, and the
slot-selection callbacks (`onSlotClick` / `onSlotSelect`), so a create-on-select
or drop-to-create also knows the chosen resource. See the **Views → TimeGridView →
DayWithResources** and **WeekWithResources** stories. (A **day-major** week
ordering — days outer, resources nested within each day — is still in progress.)

## Locking individual events

Use `draggableAccessor` to decide which events may be **dragged**, and
`resizableAccessor` to decide which may be **resized** — each takes a field name or a
function. An event that returns `false` can't be picked up (or shows no resize handles):

```tsx
<CalendarProvider draggableAccessor={(e) => !e.locked} resizableAccessor={(e) => !e.fixed} ... />
```

## Promoting a timed event to all-day

Drag any **timed** event from the time grid and drop it onto the **all-day row** —
the row that sits above the time slots. The event becomes a whole-day event on the
dropped day: `onEventDrop` fires with `allDay: true` and the event's date shifted to
that day (the time-of-day is preserved in `start`/`end` for your reference, but the
`allDay` flag is what signals the change):

```tsx
onEventDrop={({ event, start, end, allDay }) => {
  // allDay is `true` when dropped onto the all-day row.
  setEvents(prev =>
    prev.map(e => e.id === event.id ? { ...e, start, end, allDay } : e)
  )
}}
```

**One-way only.** You can promote a timed event to all-day, but you **cannot** drag
an all-day event down into the time grid. An all-day event has no natural time of day
to snap to; the right pattern is to let the user edit it in a dialog. The all-day row
ignores drops from all-day events.

See the **Views → TimeGridView → TimedToAllDayPromotion** story for a live demo.

## Not built yet

Move, resize, drop-from-outside, drag-out, keyboard drag, and timed→all-day
promotion are all wired for both the **month** and **time-grid** views today, and
the **day** and **week** views support **resource columns** (resource-major). The
remaining follow-ups: a **day-major** week resource ordering and a dedicated
`@big-calendar/react/dnd` entry point.
