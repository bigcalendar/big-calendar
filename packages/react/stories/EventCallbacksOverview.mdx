# Event Callbacks

Big Calendar fires callbacks when a user interacts with an event block — clicking, double-clicking, right-clicking, or middle-clicking. These fire on **any** view and do **not** require `selectable` to be enabled. They are completely independent of slot selection.

## The callbacks

### `onEventClick`

Fires when the user **single-clicks** an event. Receives the raw event object from your `events` array — the exact same value you passed in, with no wrapping.

```tsx
<CalendarProvider
  onEventClick={(event) => {
    setEditingEvent(event)
    setDrawerOpen(true)
  }}
>
```

### `onEventDoubleClick`

Same as `onEventClick`, fires on a **double-click**.

```tsx
<CalendarProvider
  onEventDoubleClick={(event) => {
    openFullEditor(event)
  }}
>
```

### `onEventRightClick`

Fires on a **right-click**. Receives two arguments: the event object and the browser `MouseEvent`. The browser's native context menu still appears unless you call `e.preventDefault()`.

```tsx
<CalendarProvider
  onEventRightClick={(event, e) => {
    e.preventDefault()  // suppress the browser context menu
    setContextMenu({ event, x: e.clientX, y: e.clientY })
  }}
>
```

### `onEventMiddleClick`

Fires on a **middle-click** (scroll-wheel button press). Same two-argument signature as `onEventRightClick`. Useful for opening an event in a new tab without disrupting the calendar.

```tsx
<CalendarProvider
  onEventMiddleClick={(event, e) => {
    e.preventDefault()
    window.open(`/events/${event.id}`, '_blank')
  }}
>
```

## Combining with slot selection

Event callbacks and slot selection coexist without any conflict. Add `selectable` to `CalendarProvider` to enable slot callbacks alongside event callbacks:

```tsx
<CalendarProvider
  selectable
  onSlotSelect={({ start, end }) => openCreateDialog({ start, end })}
  onEventClick={(event) => openEditDialog(event)}
>
```
