# createCalendarStore

`createCalendarStore` is the framework-agnostic heart of Big Calendar. It is a plain JavaScript function that returns a fully reactive calendar store — no React, no DOM, no rendering. If you are building a Big Calendar integration for a framework other than React, or if you want to drive the calendar from outside your component tree, this is the function you work with directly.

If you are using React, you do not call this yourself. `CalendarProvider` calls it for you and connects the result to React's rendering system automatically.

```ts
import { createCalendarStore } from '@big-calendar/core'
import { createTemporalLocalizer } from '@big-calendar/localizer-temporal'

const localizer = await createTemporalLocalizer({ locale: 'en-US', timeZone: 'UTC' })

const store = createCalendarStore({
  localizer,
  events: myEvents,
  date: '2026-06-24',
  view: 'week',
})

// Read reactive state
console.log(store.view.value)   // 'week'
console.log(store.range.value)  // { days: [...], firstVisibleDay: '...', lastVisibleDay: '...' }

// Trigger actions
store.navigate({ direction: 'NEXT' })
store.setView({ view: 'month' })

// Clean up when done
store.destroy()
```

> Every store is isolated. Creating two stores creates two completely independent calendars with no shared state between them.

## Configuration

All options are passed as a single object to `createCalendarStore`. Only `localizer` is required.

### Required

<table>
  <thead><tr><th>Option</th><th>Type</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>localizer</code></td>
      <td><code>LocalizerContract</code></td>
      <td>
        The engine that handles all date formatting and arithmetic. Big Calendar never touches
        JavaScript's built-in <code>Date</code> object directly — the localizer does all of that
        work. Create one with <code>createTemporalLocalizer</code> or <code>createLuxonLocalizer</code>
        before passing it here. Every derived value in the store recomputes when a new localizer
        is swapped in via <code>store.setLocalizer</code>, so runtime locale or timezone switching
        works without remounting.
      </td>
    </tr>
  </tbody>
</table>

### Data

These options supply the calendar's content. The store reads what you pass in but never modifies it.

<table>
  <thead><tr><th>Option</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>events</code></td>
      <td><code>TEvent[]</code></td>
      <td><code>[]</code></td>
      <td>
        Your foreground event objects. Each one needs at minimum a start date, an end date, and a
        title — but you can use any field names you like and tell the store about them via
        <code>accessors</code>. After the store is created, update the list at any time by calling
        <code>store.setEvents(&#123; events: nextEvents &#125;)</code>.
      </td>
    </tr>
    <tr>
      <td><code>backgroundEvents</code></td>
      <td><code>TEvent[]</code></td>
      <td><code>[]</code></td>
      <td>
        Background events render as coloured bands behind foreground events. Clicks and drags pass
        straight through them — they are visual only. Good for availability windows, focus blocks,
        or blocked-off periods. Update via <code>store.setBackgroundEvents</code>.
      </td>
    </tr>
    <tr>
      <td><code>resources</code></td>
      <td><code>TResource[]</code></td>
      <td><em>none</em></td>
      <td>
        Resource objects for multi-resource time grids (e.g. rooms or staff). When supplied,
        each day column in the time grid is split into one sub-column per resource. Update via
        <code>store.setResources</code>.
      </td>
    </tr>
  </tbody>
</table>

### Time grid options

These control the Week, Work Week, and Day views. They have no effect on the Month or Agenda view.

<table>
  <thead><tr><th>Option</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>step</code></td>
      <td><code>number</code></td>
      <td><code>30</code></td>
      <td>
        The size of each time slot in minutes. <code>30</code> produces slots at "9:00", "9:30",
        "10:00", and so on. Set to <code>15</code> for quarter-hour precision or <code>60</code>
        for full-hour slots. The store exposes the resolved value as <code>store.step</code>.
      </td>
    </tr>
    <tr>
      <td><code>timeslots</code></td>
      <td><code>number</code></td>
      <td><code>2</code></td>
      <td>
        How many individual slot rows appear under each labelled time. At <code>step=30</code>
        and <code>timeslots=2</code> you get two rows per half-hour — a lighter divider line
        splits each block visually. Set to <code>1</code> to remove subdivisions.
      </td>
    </tr>
    <tr>
      <td><code>min</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>Start of day (midnight)</td>
      <td>
        The earliest time shown in the time grid. <code>&#123; hour: 8 &#125;</code> starts
        the grid at 8:00 AM. Events that begin before this time are still rendered; they are
        clipped at the top edge.
      </td>
    </tr>
    <tr>
      <td><code>max</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>End of day</td>
      <td>
        The latest time shown. <code>&#123; hour: 23 &#125;</code> ends the grid at 11:00 PM.
        Passing <code>&#123; hour: 0 &#125;</code> means end of day (full 24-hour window).
        Events that run past this boundary are clipped at the bottom edge.
      </td>
    </tr>
    <tr>
      <td><code>scrollToTime</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>Current time</td>
      <td>
        The wall-clock time the time grid scrolls to on initial load. When omitted, the grid
        scrolls to the current time in the calendar's configured time zone, keeping "now" in
        view automatically. Pass a fixed value like <code>&#123; hour: 8 &#125;</code> to
        always open at 8:00 AM. The store exposes this as <code>store.scrollToTime</code> for
        framework adapters to act on at mount.
      </td>
    </tr>
    <tr>
      <td><code>dayLayoutAlgorithm</code></td>
      <td><code>'overlap' | 'no-overlap' | Function</code></td>
      <td><code>'overlap'</code></td>
      <td>
        Controls how events that share the same time slot are packed side-by-side in a column.
        <ul>
          <li><code>'overlap'</code> — groups time-close events and lets them share column width, Google Calendar style.</li>
          <li><code>'no-overlap'</code> — gives every time-touching event its own lane so nothing overlaps visually.</li>
        </ul>
        You can also pass a custom function that receives the events and returns layout boxes.
      </td>
    </tr>
    <tr>
      <td><code>showMultiDayTimes</code></td>
      <td><code>boolean</code></td>
      <td><code>false</code></td>
      <td>
        By default, events that span more than one day appear in the all-day strip above the
        time grid. Set this to <code>true</code> to render them in the time columns at their
        actual start and end times instead. Useful for overnight shifts.
      </td>
    </tr>
    <tr>
      <td><code>showAllEvents</code></td>
      <td><code>boolean</code></td>
      <td><code>false</code></td>
      <td>
        The all-day strip above the time grid is capped at two rows by default; a
        <strong>+ N more</strong> indicator appears when there are more. Set this to
        <code>true</code> to remove that cap and show every event regardless of how tall
        the strip grows.
      </td>
    </tr>
    <tr>
      <td><code>resourceLayout</code></td>
      <td><code>'resource' | 'day'</code></td>
      <td><code>'resource'</code></td>
      <td>
        Only meaningful when <code>resources</code> is supplied.
        <ul>
          <li><code>'resource'</code> — one column block per resource, spanning all visible days. Good for comparing resources side-by-side.</li>
          <li><code>'day'</code> — one column block per day, with each resource as a sub-column within it. Good for a daily schedule broken down by resource.</li>
        </ul>
      </td>
    </tr>
  </tbody>
</table>

### Accessors and utilities

<table>
  <thead><tr><th>Option</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>accessors</code></td>
      <td><code>Partial&lt;Accessors&gt;</code></td>
      <td>Standard field names</td>
      <td>
        By default the store reads <code>title</code>, <code>start</code>, <code>end</code>,
        <code>allDay</code>, <code>resource</code>, <code>id</code>, <code>tooltip</code>,
        <code>type</code>, and <code>resourceType</code> from each event or resource object,

          start: (event) => event.startsAt,

    </tr>
    <tr>
      <td><code>getNow</code></td>
      <td><code>() =&gt; string</code></td>
      <td>Current UTC instant</td>
      <td>
        A function that returns the current moment as an ISO string. The store calls this to
        decide where to draw the "today" highlight and the now-indicator line. Override it to
        pin the calendar to a fixed point in time — useful in tests and in Storybook so the
        layout renders identically every time.
      </td>
    </tr>
  </tbody>

### Selection

Slot selection is how users click or drag on empty time to indicate they want to create a new event. It is disabled by default.

<table>
  <thead><tr><th>Option</th><th>Type</th><th>Default</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>selectable</code></td>
      <td><code>boolean | 'ignoreEvents'</code></td>
      <td><code>false</code></td>
      <td>
        Enables slot selection. <code>true</code> lets users click and drag anywhere in the
        grid — including on top of existing events — to make a selection.
        <code>'ignoreEvents'</code> enables selection but prevents clicks directly on event
        blocks from starting a selection; those clicks fall through to the event click handler
        instead.
      </td>
    </tr>
    <tr>
      <td><code>longPressThreshold</code></td>
      <td><code>number</code></td>
      <td><code>500</code></td>
      <td>
        On touch screens, how long (in milliseconds) a finger must be held still before a slot
        selection begins. A shorter touch is treated as a tap. Mouse and stylus inputs are not
        affected. Decrease this if the long-press feels slow.
      </td>
    </tr>
  </tbody>
</table>

## Callbacks

Callbacks are optional functions you supply in the config. The store calls them when specific things happen — a user navigates to a new date, clicks an event, drops a drag. The store never mutates your data itself; callbacks are how you find out what the user did so you can update your own state.

### Navigation

<table>
  <thead><tr><th>Option</th><th>Signature</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>onNavigate</code></td>
      <td><code>(&#123; date, view &#125;) =&gt; void</code></td>
      <td>
        Fires after the focus date changes via <code>store.navigate</code>. Receives the new
        date as an ISO string and the current view key.
      </td>
    </tr>
    <tr>
      <td><code>onView</code></td>
      <td><code>(&#123; view &#125;) =&gt; void</code></td>
      <td>
        Fires after the active view changes via <code>store.setView</code>. Receives the new
        view key.
      </td>
    </tr>
    <tr>
      <td><code>onRangeChange</code></td>
      <td><code>(&#123; range: &#123; start, end &#125;, view &#125;) =&gt; void</code></td>
      <td>
        Fires whenever the visible date range changes (date or view change), but not on the
        initial creation. <code>start</code> and <code>end</code> are full datetime strings
        covering the start of the first visible day and the end of the last visible day.
        Useful for fetching only the events that fall in the currently visible window.
      </td>
    </tr>
    <tr>
      <td><code>onDrillDown</code></td>
      <td><code>(&#123; date, view &#125;) =&gt; void</code></td>
      <td>
        When supplied, the store hands full control of drilldown to you and will <em>not</em>
        change its own view or date. Receives the target date and the view the store would have
        navigated to. Use this to drive your own router instead of letting the store navigate.
      </td>
    </tr>
  </tbody>
</table>

### Drag and drop

These fire when the user drops an event in a new position. The store fires the callback with proposed new bounds; it never mutates <code>events</code> itself. Apply the change to your own state (optimistic update) and revert on a failed save.

<table>
  <thead><tr><th>Option</th><th>Signature</th><th>Description</th></tr></thead>
  <tbody>
    <tr>
      <td><code>onEventDrop</code></td>
      <td><code>(&#123; event, start, end, allDay, resourceId?, backgroundEvents? &#125;) =&gt; void</code></td>
      <td>
        Fires when a dragged event is dropped at a new position. Receives the original event
        plus its proposed new <code>start</code> and <code>end</code> as ISO strings. The
        original duration is preserved.
      </td>
    </tr>
    <tr>
      <td><code>onEventResize</code></td>
      <td><code>(&#123; event, start, end, allDay, resourceId?, backgroundEvents? &#125;) =&gt; void</code></td>
      <td>
        Same shape as <code>onEventDrop</code>, fires when a resize handle is dragged and
        dropped. The un-dragged edge is unchanged; the duration is clamped to at least one slot.
      </td>
    </tr>
    <tr>
      <td><code>onDropFromOutside</code></td>
      <td><code>(&#123; start, end, allDay, resourceId?, backgroundEvents? &#125;) =&gt; void</code></td>
      <td>
        Fires when an item dragged from <em>outside</em> the calendar (a palette or unscheduled
        list) is dropped on a slot. There is no existing event yet — you create one and add it
        to your own <code>events</code> array.
      </td>
    </tr>
    <tr>
      <td><code>onEventDragStart</code></td>
      <td><code>(&#123; event &#125;) =&gt; void</code></td>
      <td>
        Fires when the user begins dragging an event out of the calendar. Pair it with a
        native drop target elsewhere in your UI to build an "unschedule" bin.
      </td>
    </tr>
  </tbody>
</table>

## What the store gives you back

`createCalendarStore` returns a `CalendarStore` object. Here is a quick map of what is on it.

### Reactive state (writable signals)

These are `Signal` objects from `@preact/signals-core`. Read the current value with `.value`; write a new one by assigning to `.value`. Any computed signal or subscriber that reads one of these will automatically update.

<table>

### Actions

Actions are methods that mutate state or fire callbacks. All take a single named-argument object.

<table>
  <thead><tr><th>Method</th><th>Description</th></tr></thead>
  <tbody>
    <tr><td><code>navigate(&#123; direction, date? &#125;)</code></td><td>Move the focus date. <code>direction</code> is <code>'PREV'</code>, <code>'NEXT'</code>, <code>'TODAY'</code>, or <code>'DATE'</code>. Pass <code>date</code> when using <code>'DATE'</code> to jump to a specific ISO string.</td></tr>
    <tr><td><code>setView(&#123; view &#125;)</code></td><td>Switch the active view.</td></tr>
    <tr><td><code>setDate(&#123; date &#125;)</code></td><td>Set the focus date directly (no navigate direction semantics).</td></tr>
    <tr><td><code>selectEvent(&#123; id &#125;)</code></td><td>Select an event by id, or pass <code>null</code> to clear. Fires <code>onEventSelect</code>.</td></tr>
    <tr><td><code>getEvent(&#123; id &#125;)</code></td><td>Look up a foreground event by its accessor id. Returns <code>undefined</code> when not found.</td></tr>
    <tr><td><code>setEvents(&#123; events &#125;)</code></td><td>Replace the foreground event list.</td></tr>
    <tr><td><code>setBackgroundEvents(&#123; events &#125;)</code></td><td>Replace the background event list.</td></tr>
    <tr><td><code>setResources(&#123; resources &#125;)</code></td><td>Replace the resource list.</td></tr>
    <tr><td><code>setLocalizer(&#123; localizer &#125;)</code></td><td>Swap the active localizer. All derived signals recompute immediately. Use for runtime locale or timezone switching.</td></tr>
    <tr><td><code>drilldown(&#123; date &#125;)</code></td><td>Trigger a drilldown on the given date (resolves target view per <code>drilldownView</code> / <code>getDrilldownView</code>).</td></tr>
    <tr><td><code>moveEvent(&#123; id, target, mode, resourceId?, promote? &#125;)</code></td><td>Drop a dragged event at a new position. Fires <code>onEventDrop</code>.</td></tr>
    <tr><td><code>resizeEvent(&#123; id, edge, target, mode?, resourceId? &#125;)</code></td><td>Commit a resize drag. Fires <code>onEventResize</code>.</td></tr>
    <tr><td><code>dropExternal(&#123; target, mode?, durationMinutes?, allDay?, start?, end?, resourceId? &#125;)</code></td><td>Commit an external (outside-calendar) drop. Fires <code>onDropFromOutside</code>.</td></tr>
    <tr><td><code>destroy()</code></td><td>Dispose internal subscriptions. Call this when you tear down the calendar instance to avoid memory leaks. Safe to call more than once.</td></tr>
  </tbody>
</table>

> There are also preview actions (`previewMove`, `previewResize`, `previewExternal`, `clearDragPreview`) and keyboard-drag actions (`grabEvent`, `grabMove`, `grabResize`, `grabCommit`, `grabCancel`) used by the DnD and accessibility layers. Framework adapter authors will need these; most application developers will not.
