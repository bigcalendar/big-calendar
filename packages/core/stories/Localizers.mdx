# Localizers

Big Calendar stores all dates as **plain text strings** in the
[RFC 3339](https://datatracker.ietf.org/doc/html/rfc3339) format — something
like `"2026-07-03T14:30:00Z"`. A string on its own is just characters though.
The calendar needs something that can _understand_ that string: add a week to it,
compare two of them, figure out which time zone offset applies, and turn it into
a human-readable label like "2:30 PM".

That something is the **localizer**.

Think of the localizer as a translator. On one side it speaks "plain date
strings." On the other side it speaks a date library (or the browser's built-in
Temporal API). You plug a localizer into `createCalendarStore` and every view,
every event, and every interaction goes through it — the rest of the calendar
never touches raw date math directly.

```
Your app                      Big Calendar core
──────────     ──────────────────────────────────────────────────────
"2026-07-03"  →  localizer  →  "add 7 days"  →  "2026-07-10"
                              "format for display" →  "Thu, Jul 3"
```

**A localizer is required.** There is no default. You must create one and pass
it when setting up the calendar.

---

## The two built-in localizers

Big Calendar ships two ready-made localizers. Both implement the same contract,
so your calendar code works identically no matter which one you choose.

### `@big-calendar/localizer-temporal` (recommended)

This localizer is built on the
[Temporal API](https://tc39.es/proposal-temporal/docs/) — the modern date and
time standard coming to the web platform. Temporal was designed specifically to
fix the historical problems with JavaScript's `Date` object (time zone handling,
arithmetic edge cases, immutability), so it is the most correct and
future-proof option.

**When to use it:** When starting a new project, or when you have no existing
date library preference. It will be the right choice for most projects.

**One thing to know:** Temporal is not yet in every browser, so this localizer
automatically loads a polyfill when it is needed. That polyfill load is
asynchronous, which is why the factory function is `async`.

#### Installation

```bash
pnpm add @big-calendar/localizer-temporal
# or: npm install @big-calendar/localizer-temporal
```

#### Basic usage

```ts
import { createTemporalLocalizer } from '@big-calendar/localizer-temporal'
import { createCalendarStore } from '@big-calendar/core'

// `await` because the polyfill may need to load first
const localizer = await createTemporalLocalizer({
  locale: 'en-US',
  timezone: 'America/New_York',
})

const store = createCalendarStore({ localizer, /* …other options */ })
```

---

### `@big-calendar/localizer-luxon`

This localizer is built on [Luxon](https://moment.github.io/luxon/), a popular
JavaScript date library. It is a great choice if Luxon is already part of your
project — your team already knows it, it is in your bundle anyway, and you can
reuse the same library in your own code and in the calendar.

**When to use it:** When your project is already using Luxon.

**One thing to know:** Luxon is a `peerDependency`, which means _you_ install
it alongside this package. The factory is synchronous — no `await` needed.

#### Installation

```bash
pnpm add @big-calendar/localizer-luxon luxon
# or: npm install @big-calendar/localizer-luxon luxon
```

#### Basic usage

```ts
import { createLuxonLocalizer } from '@big-calendar/localizer-luxon'
import { createCalendarStore } from '@big-calendar/core'

// synchronous — no await
const localizer = createLuxonLocalizer({
  locale: 'en-US',
  timezone: 'America/New_York',
})

const store = createCalendarStore({ localizer, /* …other options */ })
```

## Configuration options

Both localizers accept the same set of options. None are required — the
localizer will pick sensible defaults from your browser environment — but most
real apps will want to set at least `locale` and `timezone` explicitly so that
the calendar renders consistently for every user, regardless of where they are.

### `locale`

A [BCP 47](https://www.ietf.org/rfc/bcp/bcp47.txt) language tag, or an
`Intl.Locale` object.

This controls how dates and times are formatted for display: which language,
which calendar system, whether months come before or after days, and so on.

```ts
// English as spoken in the United States
createLuxonLocalizer({ locale: 'en-US' })

// French as spoken in France
createLuxonLocalizer({ locale: 'fr-FR' })

// English with a 24-hour clock preference explicitly requested
createLuxonLocalizer({ locale: 'en-US-u-hc-h23' })
```

**Default:** the locale reported by the user's browser (`Intl.DateTimeFormat().resolvedOptions().locale`).

---

### `timezone`

An [IANA time zone](https://www.iana.org/time-zones) name — the kind of string
you find in a database like `'America/New_York'`, `'Europe/London'`, or
`'Asia/Tokyo'`.

This tells the calendar which time zone to use for all wall-clock math: when
does a day start, what time does an event appear at, and so on.

```ts
createLuxonLocalizer({ timezone: 'America/Chicago' })
```

**Default:** the time zone reported by the user's browser (`Intl.DateTimeFormat().resolvedOptions().timeZone`).

> **Tip for server-side rendering:** browser defaults are `undefined` on the
> server. Always supply an explicit `timezone` (and usually an explicit `locale`)
> when rendering on the server so the output matches what the client will render.

---

### `output`

Controls the format of date strings the localizer _writes back out_ after doing
math on them. This affects the strings you receive in callbacks like
`onEventDrop`, and the strings stored internally.

| Value | What it looks like | When to use it |
|---|---|---|
| `'utc'` (default) | `2026-07-03T16:00:00Z` | Most apps. UTC is unambiguous and easy to store. |
| `'offset'` | `2026-07-03T12:00:00-04:00` | When you need the local wall-clock offset preserved in the string. |

```ts
createLuxonLocalizer({ timezone: 'America/New_York', output: 'offset' })
// strings this localizer writes will look like: 2026-07-03T12:00:00-04:00
```

---

### `extendedZone`

When `true`, the localizer appends the IANA time zone name in square brackets
at the end of every date string it writes. This is the
[RFC 9557](https://www.rfc-editor.org/rfc/rfc9557) format (sometimes called
"bracketed IANA notation").

```ts
createLuxonLocalizer({ timezone: 'America/New_York', extendedZone: true })
// strings look like: 2026-07-03T16:00:00Z[America/New_York]
```

This format preserves the intended time zone even when the UTC offset is
ambiguous (e.g. during a daylight saving transition). It is useful when you
store dates in a database and want to round-trip the time zone without loss.

**Default:** `false`.

---

### `firstDayOfWeek`

Which day the calendar week starts on. Uses ISO-8601 numbering: `1` = Monday,
`7` = Sunday.

```ts
// Start the week on Monday (common in Europe)
createLuxonLocalizer({ firstDayOfWeek: 1 })

// Start the week on Sunday (common in North America)
createLuxonLocalizer({ firstDayOfWeek: 7 })
```

**Default:** derived from your `locale`'s regional week data. For `'en-US'` that
is Sunday (`7`); for `'en-GB'` it is Monday (`1`).

---

### `formats`

An optional map letting you override the `Intl.DateTimeFormat` option sets used
for specific display roles. You only need to provide the roles you want to
change — the rest keep their built-in defaults.

The available roles are:

| Role | Used for | Default options |
|---|---|---|
| `date` | Generic date display | `{ year: 'numeric', month: 'numeric', day: 'numeric' }` |
| `time` | Generic time display | `{ hour: 'numeric', minute: '2-digit' }` |
| `datetime` | Date + time together | year, month, day, hour, minute |
| `weekday` | Full weekday name | `{ weekday: 'long' }` |
| `weekdayShort` | Abbreviated weekday | `{ weekday: 'short' }` |
| `dayOfMonth` | Just the day number | `{ day: 'numeric' }` |
| `monthDay` | Month + day | `{ month: 'long', day: 'numeric' }` |
| `monthYear` | Month + year | `{ month: 'long', year: 'numeric' }` |
| `monthHeader` | Month view toolbar heading | `{ month: 'long', year: 'numeric' }` |
| `dayHeader` | Day view toolbar heading | weekday, month, day |
| `dayColumnHeader` | Per-column heading in week view | `{ weekday: 'short', day: '2-digit' }` |
| `agendaDate` | Date column in agenda view | weekday short, month short, day |
| `agendaTime` | Time column in agenda view | `{ hour: 'numeric', minute: '2-digit' }` |
| `timeGutter` | Left-hand time labels in time grid | `{ hour: 'numeric', minute: '2-digit' }` |
| `selectRange` | Tooltip shown while dragging a selection | `{ hour: 'numeric', minute: '2-digit' }` |

Each value is a plain `Intl.DateTimeFormat` options object. For example, to show
24-hour time in the time gutter:

```ts
createLuxonLocalizer({
  locale: 'en-US',
  timezone: 'America/New_York',
  formats: {
    timeGutter: { hour: '2-digit', minute: '2-digit', hour12: false },
    agendaTime: { hour: '2-digit', minute: '2-digit', hour12: false },
  },
})
```

## Which localizer should I pick?

| Situation | Use |
|---|---|
| Starting fresh with no date library | `localizer-temporal` |
| Project already uses Luxon | `localizer-luxon` |
| Need to support very old browsers where polyfill size matters | Either — both polyfill only what is needed |
| Storing dates in a database and need the time zone in the string | Either, with `extendedZone: true` |
| Need to build a custom integration with a different date library | [Write a custom localizer](/?path=/docs/core-custom-localizer--docs) |

## Input: what strings the localizer accepts

The localizer is flexible about what it reads in. All of these are valid inputs:

```
2026-07-03                        ← date only (treated as midnight in your timezone)
2026-07-03T16:00:00Z              ← UTC instant
2026-07-03T12:00:00-04:00         ← instant with explicit offset
2026-07-03T16:00:00Z[America/New_York]  ← RFC 9557 with IANA bracket
```

All four represent the same moment in time (noon in New York, 4 PM UTC, on July
3rd 2026). The localizer normalizes any of these to its configured timezone on
the way in.

---

## Output: what strings the localizer writes

What you get back depends on the `output` and `extendedZone` settings you
chose:

| `output` | `extendedZone` | Example output |
|---|---|---|
| `'utc'` (default) | `false` (default) | `2026-07-03T16:00:00Z` |
| `'utc'` | `true` | `2026-07-03T16:00:00Z[America/New_York]` |
| `'offset'` | `false` | `2026-07-03T12:00:00-04:00` |
| `'offset'` | `true` | `2026-07-03T12:00:00-04:00[America/New_York]` |
