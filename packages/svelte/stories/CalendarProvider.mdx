# CalendarProvider

`CalendarProvider` is the required outer wrapper for every Big Calendar. Think of it as the brain of the whole calendar — it holds the active view, the focus date, the event list, and all of the settings. Every component or composable you use inside it reads from that shared state automatically.

In Svelte, it works through `setContext` / `getContext`: `CalendarProvider` calls `setContext()` to share the calendar store, and every composable inside (`useMonthView`, `useTimeGridView`, etc.) calls `getContext()` to read it. You never pass props down by hand.

```svelte
<script>
  import { CalendarProvider, Calendar } from '@big-calendar/svelte'
  import { createTemporalLocalizer } from '@big-calendar/localizer-temporal'
  import '@big-calendar/styles/index.css'

  const localizer = await createTemporalLocalizer({ locale: 'en-US', timeZone: 'UTC' })
  const events = [
    { id: '1', title: 'Team standup', start: '2026-06-15T09:00:00Z', end: '2026-06-15T09:30:00Z' }
  ]
</script>

<CalendarProvider {localizer} {events} defaultDate="2026-06-15">
  <Calendar />
</CalendarProvider>
```

## Date and view

### Uncontrolled (calendar manages its own state)

Pass these to set the initial values. The calendar handles navigation on its own after that.

<table>
  <thead>
    <tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><code>defaultView</code></td>
      <td><code>ViewKey</code></td>
      <td><code>'month'</code></td>
      <td>
        Which view the calendar opens on. Valid values: <code>'month'</code>,
        <code>'week'</code>, <code>'work_week'</code>, <code>'day'</code>,
        <code>'agenda'</code>. The user can switch views with the toolbar after that.
      </td>
    </tr>
    <tr>
      <td><code>defaultDate</code></td>
      <td><code>string</code></td>
      <td>Today</td>
      <td>
        The date the calendar focuses on when it first loads, as an ISO 8601 string
        (for example <code>"2026-06-15"</code>). The user can navigate away with Prev / Next.
      </td>
    </tr>
  </tbody>
</table>

### Controlled (your state drives the calendar)

Use these when your application needs to programmatically control which view or date is shown
— for example, from a URL-driven route or a custom mini-calendar.

```svelte
<script>
  let view = $state('week')
  let date = $state('2026-06-15')
</script>

<CalendarProvider
  {localizer}
  {events}
  {view}
  {date}
  onView={(v) => (view = v)}
  onNavigate={({ date: d }) => (date = d)}
>
  <Calendar />
</CalendarProvider>
```

> **Svelte idiom.** Pass `view` and `date` as regular props and respond to `onView` and
> `onNavigate` to update your `$state` variables. There is no two-way binding shorthand for
> these — the callback pattern matches the other adapters.

<table>
  <thead>
    <tr><th>Prop</th><th>Type</th><th>Description</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><code>view</code></td>
      <td><code>ViewKey</code></td>
      <td>
        The active view. When you pass this, the calendar will always show whatever view your
        state holds. Wire <code>onView</code> to update your state when the user switches
        views via the toolbar.
      </td>
    </tr>
    <tr>
      <td><code>date</code></td>
      <td><code>string</code></td>
      <td>
        The focus date as an ISO 8601 string. Wire <code>onNavigate</code> to update your
        state when the user navigates with the toolbar.
      </td>
    </tr>
  </tbody>
</table>

## Time grid props

These props control the Week, Work Week, and Day views. They have no effect on the Month or Agenda view.

<table>
  <thead>
    <tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><code>step</code></td>
      <td><code>number</code></td>
      <td><code>30</code></td>
      <td>
        The size of each time slot in minutes. <code>30</code> means slots are "9:00", "9:30",
        "10:00", and so on. Set to <code>15</code> for quarter-hour slots or <code>60</code>
        for one-hour slots.
      </td>
    </tr>
    <tr>
      <td><code>timeslots</code></td>
      <td><code>number</code></td>
      <td><code>2</code></td>
      <td>
        How many individual slot rows appear under each labelled time. At <code>step=30</code>
        and <code>timeslots=2</code> you get two rows per half-hour (a light divider line
        splits each block). Setting this to <code>1</code> removes the subdivisions.
      </td>
    </tr>
    <tr>
      <td><code>min</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>Start of day</td>
      <td>
        The earliest time shown in the time grid. <code>hour</code> is 0–23;
        <code>minute</code> defaults to <code>0</code> when omitted. For example,
        <code>min={`{ hour: 8 }`}</code> starts the grid at 8:00 AM.
      </td>
    </tr>
    <tr>
      <td><code>max</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>End of day</td>
      <td>
        The latest time shown. <code>max={`{ hour: 23 }`}</code> ends the grid at 11:00 PM.
        <code>max={`{ hour: 0 }`}</code> means end of day (full 24-hour grid).
      </td>
    </tr>
    <tr>
      <td><code>scrollToTime</code></td>
      <td><code>&#123; hour: number; minute?: number &#125;</code></td>
      <td>Current time</td>
      <td>
        The wall-clock time the time grid scrolls to when it first loads. When omitted, the
        grid scrolls to the current time. Pass a fixed value like
        <code>scrollToTime={`{ hour: 8 }`}</code> to always open at 8:00 AM.
      </td>
    </tr>
    <tr>
      <td><code>dayLayoutAlgorithm</code></td>
      <td><code>'overlap' | 'no-overlap' | Function</code></td>
      <td><code>'overlap'</code></td>
      <td>
        Controls how events that share the same time slot are packed side-by-side in a day
        column. <code>'overlap'</code> looks like Google Calendar; <code>'no-overlap'</code>
        gives every time-touching event its own lane.
      </td>
    </tr>
    <tr>
      <td><code>showAllEvents</code></td>
      <td><code>boolean</code></td>
      <td><code>false</code></td>
      <td>
        The all-day strip above the time grid is capped at two rows by default. Set
        <code>showAllEvents</code> to <code>true</code> to remove that cap.
      </td>
    </tr>
    <tr>
      <td><code>showMultiDayTimes</code></td>
      <td><code>boolean</code></td>
      <td><code>false</code></td>
      <td>
        By default, events that span more than one day appear in the all-day strip. Set this
        to <code>true</code> to render them in the time columns instead.
      </td>
    </tr>
  </tbody>
</table>

## Drilldown

"Drilling down" is what happens when a user clicks a day number in the month grid or a column heading in the week grid — the calendar zooms in to a single-day view for that date.

<table>
  <thead>
    <tr><th>Prop</th><th>Type</th><th>Default</th><th>Description</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><code>drilldownView</code></td>
      <td><code>ViewKey | null</code></td>
      <td><code>'day'</code></td>
      <td>
        Which view to land on after a drilldown. Pass <code>null</code> to disable drilldown
        entirely — day numbers become plain text with no click handler.
      </td>
    </tr>
    <tr>
      <td><code>getDrilldownView</code></td>
      <td><code>Function</code></td>
      <td><em>none</em></td>
      <td>
        A function called with the clicked date and the current view. Return a view key to
        navigate there, or <code>null</code> to block the drilldown for that specific date.
      </td>
    </tr>
    <tr>

      <td><code>Function</code></td>
      <td><em>none</em></td>

        <em>not</em> change the view or date itself.

## Accessors

By default, Big Calendar looks for `title`, `start`, `end`, `allDay`, `resource`, `type`, and `resourceType` fields on each event or resource object. If your data uses different field names, supply an `accessors` object to override the defaults.

```svelte
<!-- Your events use 'name' and 'startsAt' / 'endsAt' instead of the defaults: -->
<CalendarProvider
  accessors={{
    title: 'name',
    start: (event) => event.startsAt,
    end:   (event) => event.endsAt,
  }}
  ...
>
```

The full list of overridable accessors: `title`, `start`, `end`, `allDay`, `resource`, `id`, `tooltip`, `resourceId`, `resourceTitle`, `type`, `resourceType`, `draggable`, `resizable`.
